# 4.8.Reporter
## 4.8. Módulo de generación de informes

El módulo Reporter constituye la capa de salida del pipeline. Su responsabilidad es transformar los hallazgos correlacionados en informes consumibles, teniendo en cuenta dos perfiles de uso diferenciados: auditores de seguridad que necesitan un documento estructurado y legible con contexto suficiente para validar y remediar cada hallazgo, y sistemas automatizados de CI/CD que consumen los resultados como datos estructurados para integrarse en flujos de revisión de código.

El módulo produce dos artefactos persistidos en disco a partir de los resultados del pipeline:

* `jsons/{contrato}/{contrato}_report.json`: datos estructurados para integración con herramientas externas.
* `jsons/{contrato}/{contrato}_report.md`: informe legible en Markdown orientado al auditor.

### 4.8.1. Función principal

## `generate_report(contract_path: str, correlator_output: dict, echidna_output: dict = None, adapter_meta: dict = None) → dict`

Es la función pública del módulo y su único punto de entrada. Recibe la ruta del contrato analizado, el diccionario devuelto por `correlate()`, y opcionalmente los resultados de `run_echidna()` y los metadatos generados por `echidna_adapter.generate()`.

Internamente construye el informe completo mediante dos funciones privadas:

* `_build_json_report`: ensambla un diccionario con las siguientes secciones:
  * `meta`: contrato analizado, ruta, fecha de generación y herramientas empleadas.
  * `summary`: totales de hallazgos, número de confirmados y detectados, hallazgos de severidad alta, `risk_score` y contadores de propiedades Echidna testables y no testables.
  * `findings`: lista completa de hallazgos correlacionados.
  * `echidna_results`: resultados del fuzzing por propiedad.
* `_build_markdown_report`: genera a partir de ese mismo diccionario un informe Markdown legible con resumen ejecutivo, tabla de hallazgos, detalle por hallazgo con la evidencia de cada herramienta, resultados de Echidna y recomendaciones de mitigación priorizadas por severidad.

`generate_report` guarda ambos artefactos en disco y devuelve el diccionario del informe JSON como valor de retorno.

### 4.8.2. Puntuación de riesgo global

Con el objetivo de proporcionar una métrica sintética del nivel de riesgo del contrato analizado, el Reporter calcula una puntuación de riesgo (`risk_score`) en una escala de 0 a 10.

La fórmula aplicada es una suma ponderada de severidad por nivel de confianza, normalizada a 10:

```text
risk_score = min( Σ (SEVERITY_WEIGHT[severity] × confidence_score) / 9, 10 )
```

Los pesos de severidad utilizados son:

| Severidad      | Peso |
| -------------- | ---- |
| high           | 10   |
| medium         | 5    |
| low            | 2    |
| informational  | 0    |

El `confidence_score` toma el valor 3 para hallazgos confirmados por ambas herramientas y el valor 2 para hallazgos detectados por una sola herramienta.

La constante normalizadora 9 se elige de modo que un contrato con tres hallazgos de severidad alta confirmados por ambas herramientas alcance la puntuación máxima: `3 × 10 × 3 = 90 → 90 / 9 = 10`.

```python
SEVERITY_WEIGHT = {"high": 10, "medium": 5, "low": 2, "informational": 0, "optimization": 0}

def _risk_score(findings: list[dict]) -> float:
    total = sum(
        SEVERITY_WEIGHT.get(f.get("severity", "informational"), 0) * f.get("confidence_score", 2)
        for f in findings
    )
    return round(min(total / 9, 10), 1)
```

### 4.8.3. Estructura del informe JSON

El informe JSON completo devuelto por `generate_report()` tiene cuatro secciones de primer nivel:

```json
{
  "meta": {
    "contract": "MultiVuln",
    "contract_path": "contracts/MultiVuln.sol",
    "generated_at": "2025-06-10T14:32:05",
    "tools": ["slither", "mythril", "echidna"]
  },
  "summary": { ... },
  "findings": [
    {
      "contract": "MultiVuln",
      "function": "kill",
      "swc_id": "SWC-106",
      "vuln_type": "unprotected_selfdestruct",
      "severity": "high",
      "confidence_score": 3,
      "status": "confirmed",
      "confirmed_by": ["mythril", "slither"],
      "lines": [12],
      "evidence": { "slither": { ... }, "mythril": { ... } }
    }
  ],
  "echidna_results": [
    {
      "swc_id": "SWC-106",
      "function": "kill",
      "detector": "suicidal",
      "echidna_testable": true,
      "echidna_status": "failed",
      "echidna_passed": false
    }
  ]
}
```

La sección `summary` del informe JSON tiene la siguiente estructura (ejemplo real del contrato `MultiVuln.sol`):

```json
{
  "total_findings": 3,
  "confirmed": 2,
  "detected_only": 1,
  "high_severity": 2,
  "risk_score": 8.3,
  "echidna_testable": 2,
  "echidna_non_testable": 1
}
```

### 4.8.4. Informes de texto en Markdown

El informe Markdown generado por `_build_markdown_report` incluye:

* Resumen ejecutivo con los metadatos del análisis y la puntuación de riesgo.
* Tabla de hallazgos correlacionados con severidad, herramientas de origen y estado (confirmado o detectado).
* Detalle individual de cada hallazgo con la evidencia original de cada herramienta y la localización afectada.
* Resultados del fuzzing con Echidna: estado de cada propiedad (passed, failed, no-testable) y advertencias sobre propiedades no testables automáticamente.
* Recomendaciones de mitigación priorizadas por severidad.

Este formato permite que el informe resultante pueda ser revisado directamente por el auditor sin necesidad de herramientas adicionales.

### 4.8.5. Integración con el pipeline

El Reporter recibe su entrada directamente del Correlator y, opcionalmente, del Runner (resultados de Echidna) y del Echidna Adapter (metadatos de propiedades generadas). Esta posición al final del pipeline garantiza que el informe refleja la visión consolidada de todas las fases anteriores.

No existen funciones públicas independientes `generate_summary` ni `export_to_sarif`: el resumen ejecutivo se construye internamente como parte de `_build_json_report`. La exportación al formato estándar SARIF 2.1.0 se plantea como línea de trabajo futuro, dado su interés para integrar EVMAudit en flujos de CI/CD.
